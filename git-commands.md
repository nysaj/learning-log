# Git/GitHubのコマンド学習記録

## ローカルリポジトリ（自分のパソコンの保管場所）の作成（最初に一度だけ）

* `git init`
    Gitで管理したいディレクトリ（フォルダ）内で最初に一度だけ実行

## ブランチ操作

* `git branch`
    現在のブランチ名を確認
* `git branch -M main`
    デフォルトのブランチ（分岐）の名前を「main」にする

## 変更内容を確認・記録

* `git status`
    今どのファイルが変更されたか、コミット（記録）の準備ができているかの状態を表示
* `git add [ファイル名]`
    ファイル名をステージングエリア（記録する対象リスト）に登録
* `git add .`
    変更したすべてのファイルを、ステージングエリア（記録する対象リスト）に登録（ドットは今いるフォルダ以下全部という意味）
* `git commit -m "[メッセージ]"`
    ステージングエリア（記録する対象リスト）に登録したファイルを、履歴としてメッセージ付きでコミット（記録）する
* `git remote -v`
    - GitHubのリモートリポジトリ（接続先）のURLを詳細表示する
    - `-v` （バーブ、verbose）は詳細表示のオプション

## リモートリポジトリ（GitHub）との連携（最初に一度だけ）

* `git remote add origin [GitHubのURL]`
    ローカルリポジトリ（自分のパソコンの保管場所）と、GitHubのリモートリポジトリ（保管場所）を紐付ける

## GitHubへ送信

* `git push -u origin main`
    ローカルリポジトリの中身（セーブデータ）をGitHubに送信（初回）
* `git push`
    コミット（記録）した変更内容をGitHubへ送信（2回目以降）

## コミットメッセージのルール

* `feat`（フィート）
    - 新しい機能を追加したとき
    - 例： `feat: Aboutページを追加`
* `fix`（フィックス）
    - バグを修正したとき
    - 例： `fix: ログインエラーを修正`
* `docs`（ドックス）
    - Markdown、README、説明書、コメントなどのドキュメントを作成・変更したとき
    - 例： `docs: Gitコマンド一覧を更新`
* `style`（スタイル）
    - プログラムの動作に影響しない見た目や書式を修正したとき
    - 例： `style: インデントを修正`
* `refactor`（リファクター）
    - プログラムの動作を変えずにコードを改善したとき
    - 例： `refactor: コントローラの処理を整理`
* `chore`（チョア）
    - アプリの機能に直接関係ない設定や開発環境を変更したとき
    - 例： `chore: .envのロケール設定を変更`

## コミットメッセージで使う表現

* `improve`（インプルーヴ）
    - 既存の内容を改善し、より良くしたときに使う
    - 例：`docs: improve Git command documentation`
